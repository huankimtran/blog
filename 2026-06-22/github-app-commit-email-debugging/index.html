The Case of the Missing Bot Avatar: Debugging GitHub App Commit Attribution

22 Jun, 2026
|

The Symptom

I noticed something off on my GitHub repositories. Commits made by my NML agent containers, powered by the nomanllc GitHub App, showed a gray default avatar instead of the familiar nomanllc[bot] badge. The commit author displayed as nomanllc[bot] in the commit log, but GitHub wasn’t linking it to the bot account. No avatar, no “Bot” label: just a plain, unverified commit.

My perfectionist sense kicked in, sending a tingly sensation down my nerves. I had to fix this 🧐!

First Attempt: The Email Domain (PR #48)

GitHub App commits are attributed via the author email. The format is:

ID+APP_NAME[bot]@users.noreply.github.com

I checked the email being used in the containers and found:

1234567+nomanllc[bot]@user.noreply.github.com

Spot the bug? The domain is user.noreply.github.com: singular user, not users. GitHub’s documentation specifies the plural form: users.noreply.github.com. A single missing s.

I fixed the domain and pushed PR #48. Confident this was the fix, I waited for the next automated commit to appear.

It didn’t work. Still the gray avatar.

Second Attempt: The Token Theory (PR #50)

Maybe the email wasn’t the only factor. Perhaps GitHub also checks whether the commit was pushed using the GitHub App’s installation token versus a personal access token. If the push used a PAT, maybe GitHub ignored the bot email?

I built a credential wiring fix in PR #50 to ensure git operations used the installation token from the GitHub App rather than any personal credentials. This involved changes to how the container sourced and exported authentication tokens.

Another automated commit. Another gray avatar. Two PRs in, and the problem remained.

The Real Root Cause (PR #51)

Time to go back to basics. I re-read the email format:

ID+APP_NAME[bot]@users.noreply.github.com

But what exactly is the ID in that format? I had been using 1234567, which is the GitHub App ID:

$ gh api /apps/nomanllc --jq .id
1234567

Note: The actual GitHub App ID has been changed in this post for privacy.

But the email format requires the Bot User ID, not the App ID. Every GitHub App registration creates a special bot user account. For the nomanllc app, that user is nomanllc[bot]:

$ gh api '/users/nomanllc[bot]' --jq .id
987654321

Note: The actual Bot User ID has also been changed.

The fix was a single line change:

- git config --global user.email "1234567+nomanllc[bot]@users.noreply.github.com"
+ git config --global user.email "987654321+nomanllc[bot]@users.noreply.github.com"

PR #51: one line, and the problem was solved.

Verification

After the fix, I verified attribution with the GitHub API:

$ gh api /repos/{owner}/{repo}/commits/{sha} --jq '.author.login'
nomanllc[bot]

Before the fix, this returned null. After, it returned nomanllc[bot]. The bot avatar appeared on GitHub, and commits were properly attributed.

The Lesson

This debugging journey had three acts:

  1. The email domain fix was necessary but insufficient. user.noreply.github.com → users.noreply.github.com was a real bug, just not the only one.

  2. The token theory was a red herring. GitHub doesn’t check push credentials for commit attribution: only the author email matters. I spent time on a fix that, while improving credential hygiene, didn’t address the root cause.

  3. The ID was wrong from the start. GitHub App ID ≠ Bot User ID. The email format needs the bot’s user ID, which is a different number entirely.

The key insight came from Josh’s blog post on GitHub Apps commit email, which clearly explains the distinction between App ID and Bot User ID. GitHub’s own documentation on this is sparse; without community resources, this would have been much harder to diagnose.

Takeaways

  • Read the email format specification carefully. Every character matters: including that s in users.
  • App ID is not Bot User ID. They’re different numbers with different purposes. The App ID identifies the app registration; the Bot User ID identifies the bot as a GitHub user.
  • Verify with the API. gh api /repos/{owner}/{repo}/commits/{sha} --jq '.author.login' tells you definitively whether GitHub attributes the commit to your bot.
  • Community resources fill documentation gaps. GitHub’s docs on bot commit emails are thin. Blog posts and forum discussions often have the practical details you need.

Debugging is rarely a straight line. Sometimes it takes three PRs to find the one line that matters. 🔍
